# Framework Template

For methodology, framework, or operating-model documents. Includes both **professional frameworks** (interview rubrics, decision frameworks, prioritisation criteria, evaluation rubrics, planning playbooks) and **personal frameworks** (life frameworks, decision-making heuristics, values articulations, operating principles for yourself).

The template adapts to both. The sections below are stable; the surface flavour shifts depending on whether the framework is for a team to apply or for the author themselves to apply.

**Spec type: spec-is-the-deliverable.** The framework spec itself is the artefact. There's no separate downstream deliverable to draft from it. The handoff is type-aware (see SKILL.md handoff section).

## Sections

| Section | What goes here |
|---|---|
| Purpose | What does this framework help its user do or decide? Stated as the user benefit, not the framework name. |
| Operational definition | If the framework rests on a contested or fuzzy concept (e.g. "good life," "high performer," "strategic alignment"), define it concretely so the rest of the framework has an anchor. |
| Principles | 3-5 (sometimes up to 7) underlying principles the framework rests on. Why these, and not others. Attribute to existing thinkers where principles map to known work. |
| Structure | The framework itself. Components, steps, quadrants, layers, or tracks, with one-line descriptions AND explicit connections between elements. Placement alone is not enough; show how the parts relate. |
| When to use | What situations this framework applies to. What it does not cover. |
| When NOT to use | Edge cases or contexts where the framework would mislead. Explicit. |
| Worked example | One or more concrete examples of the framework applied. Without this the framework is abstract; with it, the framework becomes legible. |
| How to adapt | Knobs the user can turn for their context. What's load-bearing, what's flexible. |
| Anti-patterns | Common misuses. How the framework gets misapplied or weaponised. |
| Sources | Existing thinkers, traditions, or works the framework draws from. Cite them. |

## Elicitation prompts

When self-shaping, one at a time:

1. What does this framework help someone do? (Specific user, specific decision.)
2. If the framework rests on a contested concept, what's your operational definition?
3. What problem are existing frameworks not solving? Why does this one need to exist?
4. What 3-5 principles is this resting on? Why these?
5. Where do these principles map to existing thinkers' work? (Naval, Housel, Stoic, Buddhist, etc., depending on domain.)
6. What's the simplest possible structure that captures the principles AND shows the connections between them?
7. Walk me through one worked example. What does using this look like in practice?
8. What are the most likely misuses?

When a template match is suspected (user said "framework", "rubric", "playbook for"), batched only if routine for this user:

> "To frame the framework: what does it help someone do, what 3-5 principles is it resting on, and what's the simplest structure that captures the connections between them?"

If the topic is exploratory (life framework, novel professional framework, first time the user has authored one of these), revert to one-question-at-a-time even though the template matches. See SKILL.md "Adaptive Elicitation."

## Standards baked in

- **Purpose stated as user benefit.** Not "the X framework," but "helps you decide Y."
- **Operational definition for fuzzy concepts.** If the framework's foundation is a contested term, define it concretely.
- **Principles before structure.** The structure is downstream of the principles. State them first.
- **Connections, not just placement.** The structure must show how elements relate, not just where they sit. Arrows, hierarchies, dependencies, governing layers, all count. Loose box arrangements are not enough.
- **Source attribution mandatory.** When principles map to existing thinkers (Naval, Housel, Aristotle, Buddhist or Stoic traditions, etc.), cite. This adds rigour and lets the reader trace.
- **Worked example mandatory.** Without it the framework is abstract and unusable.
- **Anti-patterns named.** Most frameworks fail not because they're wrong but because they're misapplied.
- **Load-bearing vs flexible explicit.** Users will modify the framework. Tell them what they can change without breaking it.

## Anti-patterns

- Frameworks that exist for their own sake, not for a user benefit
- Structure presented before principles, leaving the structure feeling arbitrary
- Loose box arrangements without connections (the "generic diagram" failure mode)
- Skipping the worked example because it feels like padding (it isn't)
- Failing to call out how the framework gets misused
- Presenting the framework as universal when it actually has limits
- Inventing principles that map cleanly to existing thinkers' work without attributing
